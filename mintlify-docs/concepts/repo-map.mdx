The repo map replaces flat file tree listings with a graph-ranked, context-adaptive view of your codebase. Read the [full story](https://proxysoul.com/blog/soulforge) for the motivation behind this approach.

## How it works

### 1. Index phase (startup)

Walk the file tree (respecting `.gitignore`). For each source file:

1. Parse with tree-sitter to extract symbols (functions, classes, interfaces, types, enums) and their signatures
2. Extract import statements and identifier references
3. Build cross-file edges: if file A references a symbol exported by file B, create an edge A → B
4. Store everything in SQLite

Indexing is incremental — files are re-indexed only when they change.

### 2. Graph phase

Run PageRank over the file-to-file edge graph. Files imported by many other files score higher. The algorithm uses a personalization vector that can be biased toward files relevant to the current conversation.

### 3. Co-change phase

Analyze recent git history to find files that are frequently edited together. This captures implicit coupling that the import graph misses — files that always change in lockstep even without direct imports.

### 4. Ranking phase (per-turn)

Each turn, the ranking is personalized based on conversation context:

- **Edited files** get the highest boost
- **Mentioned files** (tool reads, grep hits) get a moderate boost
- **Active editor file** gets a small boost
- **Co-change partners** of context files are boosted proportionally

Additional signals like full-text search matches on conversation terms and graph proximity further refine the ranking.

### 5. Rendering phase

The ranked files are rendered into the system prompt within a token budget that scales with conversation length — more map at the start when the AI needs orientation, less as context is established.

Each file block shows exported symbols with signatures, blast radius (how many files depend on it), and optional semantic summaries:

```
src/core/agents/agent-bus.ts (→12)
  +AgentBus — Shared coordination bus for parallel subagent communication
  +SharedCache
```

- `+` = exported symbol
- `(→12)` = blast radius (12 files depend on this one)
- `[NEW]` = file appeared since last render

### 6. Semantic summaries

Top symbols (by PageRank) get one-line LLM-generated descriptions, cached by file modification time. When a file is edited, summaries are regenerated. Configurable via `semanticSummaries` in config (`"ast"`, `"llm"`, or `"off"`).

## Real-time updates

When the agent edits a file, the repo map re-indexes it automatically. The next system prompt gets updated rankings reflecting the change. File reads also influence the personalization — reading a file boosts it and its neighbors in the next ranking.

## Language support

Convention-based visibility detection for 30+ languages:

| Convention | Languages | Rule |
|---|---|---|
| Name-based | Go | Capitalized = public |
| Name-based | Python, Dart | No `_` prefix = public |
| Keyword-based | Rust, Zig | `pub` keyword |
| Keyword-based | Java, Kotlin, Swift, C#, Scala | Not `private` = public |
| Keyword-based | Elixir | `def` (not `defp`) = public |
| File-based | C, C++, Objective-C | Header file = public |
| Default public | Ruby, Lua, Bash, OCaml | All top-level symbols treated as public |

### Monorepo support

The repo map indexes files within the working directory. In monorepo setups, cross-package imports resolve as external dependencies. Co-change analysis works across the full git history regardless of package boundaries. The `project` tool handles workspace discovery separately via `project(action: "list")`.
