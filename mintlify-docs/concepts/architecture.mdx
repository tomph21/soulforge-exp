## Three-tier agent architecture

| Agent | Purpose |
|-------|---------|
| **Forge** | Main orchestrator — plans, dispatches, responds to the user |
| **⚡ Spark** | Read-only research and code analysis. Shares the forge's cache for efficiency |
| **🔥 Ember** | File edits, refactoring, implementation. Uses its own model and context |
| **WebSearch** | Multi-step web research with scraping and synthesis |

## AgentBus

Coordination layer for parallel subagents:

- **File cache** — first reader caches the content; concurrent readers share it. Edits invalidate the cache.
- **Tool result cache** — read-only tool results are cached across agents and dispatches.
- **Edit mutex** — concurrent edits to the same file are serialized. First editor owns the file; second gets a warning.
- **Findings** — agents share discoveries in real-time. One agent's finding influences the next agent within a step or two.

## Step lifecycle

Each agent step goes through preparation that handles context management:

- **Peer findings** — new discoveries from other agents are injected
- **Semantic pruning** — stale reads and old tool results are summarized to save context
- **Budget management** — agents are guided to wrap up as they approach their context budget
- **Edit preservation** — file edits are always kept in full, never pruned

## Result pipeline

When subagents finish, their results are aggregated and compressed before the parent Forge agent sees them. Multi-agent dispatches combine all results and findings into a single summary.

## Agent quality

- **Schema enforcement** — dispatch tasks require target file paths, validated before any agent runs
- **Tier routing** — tasks are classified and routed to the appropriate model tier (spark for read-only, ember for edits)
- **De-sloppify pass** — optional cleanup agent reviews code edits in fresh context to catch sloppy patterns
- **Verify pass** — optional adversarial review checks correctness after code agents finish

All features can be toggled via `/agent-features` or `agentFeatures` in config.

## Intelligence router

Routes code intelligence operations to the best available backend.

| Backend | Tier | Capabilities |
|---------|------|-------------|
| **LSP** | 1 | definitions, references, rename, diagnostics, code actions, call hierarchy, type info, formatting |
| **ts-morph** | 2 | TypeScript/JavaScript --AST definitions, references, rename, extract, unused detection |
| **tree-sitter** | 2 | 33+ languages --symbol extraction, imports/exports, scopes, outlines |
| **regex** | 3 | Universal fallback --symbol search, simple definitions, import patterns |

For each operation, the router tries backends in tier order. If tier 1 returns null or throws, tier 2 is tried, then tier 3.

### LSP integration

Dual-backend architecture — the agent always has LSP access regardless of editor state:

- **Neovim bridge**: When the editor is open, routes LSP requests through Neovim's running servers. Zero startup cost.
- **Standalone client**: When the editor is closed, spawns LSP servers directly. Full protocol support.
- **Multi-language warmup**: On boot, detects project languages and spawns LSP servers for all of them in parallel.

## Web search

Web search supports Brave Search API (with `BRAVE_API_KEY`) and DuckDuckGo as fallback. When a web search model is configured via the task router, the `web_search` tool spawns a multi-step agent that can run multiple queries, follow URLs, and synthesize a structured summary with citations. Results are cached.

## Context manager

Assembles the system prompt from multiple sources:

1. **Mode instructions** --persona-specific behavior
2. **Project info** --detected toolchain, package.json metadata
3. **Git context** --branch, recent commits, dirty files
4. **Repo map** --PageRank-ranked file/symbol view
5. **Memory** --persistent project/global memory from SQLite
6. **Forbidden files** --security patterns that block AI access
7. **Outside-CWD gating** --write tools require confirmation for paths outside the project
8. **Skills** --loaded skill instructions

See also: [Multi-agent dispatch](/agents/dispatch) for AgentBus coordination details, and [User steering](/agents/steering) for mid-execution redirection.

## UI layer

Built on OpenTUI (React reconciler for terminal UIs) with Zustand for state management. Lock-in mode (`/lock-in`) hides agent narration and shows only tool activity + final answer.
