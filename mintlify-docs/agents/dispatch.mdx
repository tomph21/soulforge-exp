The AgentBus runs multiple AI agents in parallel without duplicate work, file conflicts, or lost findings.

## How it works

When you dispatch multiple tasks, SoulForge spawns agents in parallel with a shared coordination layer:

- **File reads are deduplicated** — the first agent to read a file caches it; others get the cached content instantly
- **Tool results are shared** — read-only tool results (grep, glob, navigate, etc.) are cached across agents
- **Edits are serialized** — concurrent edits to the same file are queued, not raced
- **Findings propagate** — one agent's discovery is visible to other agents within a step or two
- **Caches persist** — warm caches carry over between dispatches in the same session

## Edit safety

The first agent to edit a file becomes its "owner". If a second agent tries to edit the same file, it receives a warning. After all agents finish, the parent Forge agent is told about any conflicts.

## Findings

Agents share discoveries in real-time without sharing context windows. When one agent discovers something relevant (e.g. "auth uses JWT with refresh tokens"), other agents see it at their next step and can adjust their approach.

## Dispatch flow

```
Forge dispatches tasks
    |
    v
Agents spawn with staggered starts
    |
    v
Each agent runs independently with shared cache access
    |
    v
Results aggregated and compressed
    |
    v
Combined result returned to Forge
```

Single-task dispatches skip coordination overhead and return the result directly.

## Spark vs Ember

| Agent | Role | When used |
|-------|------|-----------|
| **⚡ Spark** | Read-only research | Explore/investigate tasks. Shares the forge's cache for efficiency |
| **🔥 Ember** | Code edits | Code tasks. Uses its own model and fresh context |

The dispatch schema's `role` field controls which tier is used. Code agents are always embers.
